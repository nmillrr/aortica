## Iteration 6 - US-006: DICOM and SCP-ECG Format Readers
- Implemented: `read_dicom()` for DICOM ECG waveform files (Supplement 30/130) and `read_scp()` for SCP-ECG binary format files
- Files changed:
  - aortica/io/dicom_reader.py (new) - read_dicom() using pydicom, extracts waveform data, channel definitions, patient demographics
  - aortica/io/scp_reader.py (new) - read_scp() with minimal SCP-ECG binary parser, handles Sections 0/1/3/6
  - aortica/io/__init__.py (modified) - exports read_dicom and read_scp
  - tests/io/test_dicom_scp_readers.py (new) - 18 tests with synthetic DICOM and SCP-ECG file generators
- Learnings for future iterations:
  - pydicom was already a core dependency in pyproject.toml â€” no new dep needed
  - DICOM ECG waveform data is in WaveformSequence[0], channels are interleaved [sample0_ch0, sample0_ch1, ...]
  - Channel sensitivity & correction factor must both be applied: physical = digital * sensitivity * correction + baseline
  - SCP-ECG is a binary format with numbered sections; Section 0 has pointers, Section 1 has patient data, Section 3 has lead definitions, Section 6 has waveform data
  - SCP-ECG section offsets in Section 0 are 1-based byte offsets from start of record
  - SCP-ECG AVM (amplitude value multiplier) is stored in nanovolts per LSB â€” divide by 1000 for ÂµV
  - SCP-ECG lead ID codes map to standard lead names (1=I, 2=II, 3=V1, etc.) per AHA conventions
  - Synthetic file generators in tests are invaluable for testing binary format parsers without real data
  - ruff auto-fix handles import sorting issues from the `typing.Any` import in scp_reader.py

  - aortica/signal/__init__.py (modified) - exports score_quality, QualityReport, LeadQuality

  - PRD.md (modified) - marked US-011 complete
- Learnings for future iterations:
  - Penalty-based scoring: each artefact deducts from 100 (flatline=40, clipping=30, wander=20, motion=20); max penalty caps at 100
  - Flatline detection uses np.diff with absolute tolerance â€” catches lead-off and constant-value segments
  - Clipping detection uses samples near signal min/max rail based on the 99.5th percentile approach
  - Baseline wander uses FFT spectral energy ratio below 0.5 Hz vs total; motion artifact uses energy above 40 Hz vs total
  - Motion artifact detector gracefully skips when Nyquist < 40 Hz (low sample rates)
  - A single flatline penalty (40) yields score 60 = 'marginal', not 'poor'; need multiple artefacts to reach 'poor' (<40)
  - QualityReport.recommendation maps directly: goodâ†’accept, marginalâ†’review, poorâ†’reject

- PRD.md â€” Marked US-111 complete

### Implementation Details
- Token bucket algorithm with configurable capacity (RPM + burst) and refill rate (RPM/60)
- 4 endpoint tiers: predict (30 RPM), report (60 RPM), admin (120 RPM), system (300 RPM)
- Path prefix matching for tier classification
- RFC 6585 compliant headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset

- In-memory backend for single-process/edge; Redis backend with Lua script for atomic operations
- Config from YAML (rate_limits.yaml), environment variables (AORTICA_RATE_LIMIT_*), or code
- Exempt lists for API keys and IPs
- IP-based fallback when no API key provided

- Rate limiter must be added before CORS middleware in the middleware stack for headers to be set correctly
